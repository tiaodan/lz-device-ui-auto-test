## 1. pytest fixture 参数名匹配

**最容易搞不懂的：测试函数的参数从哪来？**

```python
def test_login_ok(login_page, test_logger, config):
    # 这些参数哪来的？没有 import，没有定义
```

**答案：参数名 = conftest.py 里的 fixture 函数名**

```python
# conftest.py 定义
@pytest.fixture
def login_page(page, language):  # ← 函数名
    return LoginPage(page, language=language)

@pytest.fixture
def test_logger():               # ← 函数名
    return get_default_logger()

# 测试函数用同名参数
def test_login_ok(login_page, test_logger):  # ← 参数名必须匹配
    # pytest 自动找到同名 fixture，执行后注入
```

**规则：**
- 参数名必须和 fixture 函数名完全一致
- pytest 自动执行 fixture，不需要手动调用
- fixture 定义在 conftest.py，所以测试文件里看不到定义

## 2. assert_visible 检查什么？

```python
assertion_test_not_logged.assert_visible(login_page.username_input, "用户名输入框")
```

**实际检查的内容：**

| 检查项 | 说明 |
|--------|------|
| 元素存在于 DOM | 能定位到 |
| 非 `display: none` | 不是隐藏样式 |
| 非 `visibility: hidden` | 不是隐藏样式 |
| 在视口内 | 不是在屏幕外 |
| `opacity > 0` | 不是透明的 |

**不检查的内容：**
- ❌ **被遮挡**（其他元素盖住它，也算"可见"）

**如果需要检查遮挡，要自己写：**
```python
box = locator.bounding_box()
center_x = box["x"] + box["width"] / 2
center_y = box["y"] + box["height"] / 2
# 检查这个坐标位置的元素是不是原元素
```

## 3. description 参数的作用

所有 assert 方法都有 description 参数：

```python
assert_visible(locator, description)      # ← 最后一个参数
assert_attribute(locator, "placeholder", "用户名", description)
```

**作用：日志输出更清晰**

```python
# 不写
assert_visible(login_page.username_input)
# 日志: 元素可见: 

# 写了
assert_visible(login_page.username_input, "用户名输入框")
# 日志: 元素可见: 用户名输入框
```

测试失败时，能一眼看出是哪个元素有问题。

## 4. 按分类写测试，不要按元素写

**推荐结构：**

```python
def test_login_page():
    # 第1类：可见性（元素都在吗？）
    assert_visible(username_input)
    assert_visible(password_input)
    
    # 第2类：属性（文本/placeholder对吗？）
    assert_attribute(username_input, "placeholder", "用户名")
    
    # 第3类：行为（能操作吗？）
    assert_enabled(username_input)
    
    # 第4类：功能（操作后结果对吗？）
    fill_credentials()
    click_login()
    assert_url_changed()
```

**好处：**
- 失败定位清晰：第1类失败→元素没了；第2类失败→配置错了
- 批量问题一眼看出：翻译漏了，所有 placeholder 在"属性"类里一起报错
- 维护简单：加新元素只需在对应分类加一行

## 5. 定位器只要够唯一就行

```html
<button class="ui-btn css-188gaox ui-btn-default ... lBtn">登录</button>
```

```python
# 只要 .lBtn 在页面唯一，就够了
login_button = page.locator(".lBtn")

# 更保险的写法（指定元素类型）
login_button = page.locator("button.lBtn")
```

**不需要写全部 class 名**，CSS 选择器 `.lBtn` 匹配包含这个类名的元素。

## 6. expect 和 assert 的区别

```python
# Playwright expect（自动重试，等待元素）
from playwright.sync_api import expect
expect(locator).to_be_visible()  # 等待直到超时，适合 UI 测试

# Python assert（立即检查，不等待）
assert locator.is_visible()  # 立即检查，失败就报错
```

**UI 测试用 expect**，因为元素可能还没渲染出来，需要等待。

## 7. 多语言配置切换

**配置文件结构：**
```
config/login/
├── login_config.yaml     # 定位器、测试数据（语言无关）
├── zh_CN.yaml            # 中文期望值
```

**测试代码自动加载对应语言：**
```python
login_page.exp["username_placeholder"]  # 自动读取对应语言的配置
```

## 9. CSS 属性选择器（*=、^=、$=）

**这不是 Playwright 特有语法，是标准 CSS3 规范，浏览器原生支持。**

```python
# 定位器例子
logo: ".signin-logo img, img[src*='logo']"
```

**逗号分隔 = "或"的关系：**
- `.signin-logo img` → class 为 signin-logo 下的 img 子元素
- `img[src*='logo']` → src 属性包含 "logo" 的 img 元素
- Playwright 会匹配任意一个，哪个能找到就用哪个

**属性选择器符号：**

| 符号 | 名字 | 例子 | 匹配 |
|------|------|------|------|
| `=` | 等于 | `[src='logo.png']` | src **完全等于** logo.png |
| `*=` | 包含 | `[src*='logo']` | src **包含** logo |
| `^=` | 开头 | `[src^='http']` | src **开头是** http |
| `$=` | 结尾 | `[src$='.png']` | src **结尾是** .png |

**举例：**

```html
<!-- 这些都能匹配 img[src*='logo'] -->
<img src="logo.png">              <!-- ✓ 包含 "logo" -->
<img src="/images/logo-main.png"> <!-- ✓ 包含 "logo" -->

<!-- 这些不匹配 -->
<img src="banner.png">            <!-- ✗ 不包含 "logo" -->
```

**验证浏览器原生支持：**

```js
// 在浏览器 Console 直接运行，不需要任何框架
document.querySelector("img[src*='logo']")
```

**谁在用这个语法：**
- CSS 样式：`img[src*='logo'] { border: 1px; }`
- Playwright：`page.locator("img[src*='logo']")`
- Selenium、jQuery 等都用

## 10. IDE 代码提示（fixture 参数加类型注解）

**问题：输入 `login_page.` 没有 attribute/method 提示**

pytest fixture 是动态注入的，IDE（VS Code Pylance）不知道返回对象的类型。

**解决：加类型注解**

```python
# 改前：没有提示
def test_login_ok(login_page, assertion_test_not_logged, test_logger, config):
    login_page.  # ← 这里没有提示

# 改后：有提示
from pages.login_page import LoginPage
from assertion_test.base import AssertionTest
from config.config_loader import Config

def test_login_ok(login_page: LoginPage, assertion_test_not_logged: AssertionTest, test_logger, config: Config):
    login_page.  # ← 现在有提示了：username_input, password_input, slider...
```

**规则：**
- 测试函数参数加 `: 类型名` 注解
- 顶部 import 对应的类
- IDE 就能提示该类的属性和方法

## 11. 配置文件命名规则

**为什么 login_config.yaml 用 `descriptions`，test_config.yaml 用 `messages`？**

两者含义不同：

| 配置 | 内容 | 类型 | 用途 |
|------|------|------|------|
| login_config.yaml `descriptions` | "用户名输入框"、"登录按钮" | 名词 | 断言参数，描述元素是什么 |
| test_config.yaml `messages` | "验证页面元素可见性"、"所有元素可见" | 句子 | 日志输出，描述做了什么/结果 |

**如果统一叫 descriptions：**
- ❌ 混淆：同一个名字，含义不同（元素 vs 步骤）
- ❌ 代码里 `config.descriptions` 不知道是哪个

**区分命名的好处：**
- ✅ 语义清晰：名字表达用途
- ✅ 代码里一眼看出：`login_page.desc["username_input"]`（元素）vs `config.messages["step_1"]`（步骤）

**配置分工总结：**

| 配置文件 | 字段 | 内容 |
|----------|------|------|
| test_config.yaml | `messages` | 测试步骤描述（step、pass、warn、fail） |
| login_config.yaml | `descriptions` | 元素描述（用户名输入框、登录按钮等） |
| zh_CN.yaml / en_US.yaml | 期望值 | placeholder 文本、按钮文本等 |

## 12. 断言测试原则：精确匹配，不要模糊匹配

**测试就是要测得最准确，不要用包含匹配（contains）留模糊空间。**

| 方法 | 匹配方式 | 推荐 |
|------|----------|------|
| `assert_text` | 精确匹配 | ✓ 推荐 |
| `assert_text_contains` | 包含匹配 | ❌ 不推荐 |
| `assert_attribute` | 精确匹配 | ✓ 推荐 |
| `assert_attribute_contains` | 包含匹配 | ❌ 不推荐 |

**为什么？**

| 精确匹配 | 包含匹配 |
|----------|----------|
| 测得准，能发现细微差异 | 模糊，可能漏掉问题 |
| 配置错了必报错 | 配置错了可能不报错 |
| 符合测试目的 | 留有模糊空间 |

**举例：**

```python
# 实际按钮文本是 "登录"
assert_text(button, "登录")           # ✓ 推荐（精确匹配）
assert_text_contains(button, "登录")  # ❌ 不推荐（模糊，"请登录"也能通过）

# 实际 alt="历正科技"
assert_attribute(img, "alt", "历正科技")  # ✓ 推荐
assert_attribute_contains(img, "alt", "历正")  # ❌ 不推荐

# src 属性也写完整路径
assert_attribute(img, "src", "/asset/logo-full.png")  # ✓ 推荐
```

**配置文件也要写完整值：**

```yaml
# zh_CN.yaml
slider_text: "请拖动滑块进行验证"  # ✓ 精确，完整文本
logo_src: "/asset/logo-full.png"  # ✓ 精确，完整路径
```

## 13. 常见问题速查

| 问题 | 答案 |
|------|------|
| 参数从哪来？ | 参数名 = conftest.py 的 fixture 函数名 |
| assert_visible 检查遮挡吗？ | 不检查 |
| assert_attribute 区分大小写吗？ | ✅ 区分大小写 |
| 用精确匹配还是包含匹配？ | **精确匹配**，不要用 contains |
| login_page. 不出提示？ | 加类型注解 `login_page: LoginPage` |
| test_logger. 不出提示？ | 加类型注解 `test_logger: StepLogger` |
| description 必须写吗？ | 不必须，但写了日志更清晰 |
| 定位器要写全部 class 吗？ | 不需要，够唯一就行 |
| expect 和 assert 用哪个？ | UI 测试用 expect（自动等待） |
| assert_attribute vs assert_text？ | attribute 检查 HTML 属性，text 检查显示文本 |
