## 一句话介绍

无人机防御系统的 UI自动化测试框架，支持三种测试类型：断言、视觉、接口。

---

## 项目背景

- **目标**：测试一个 Web UI 系统（无人机防御控制台）
- **技术栈**：Playwright（UI自动化）+ pytest（测试框架）+ GraphQL（API）
- **特点**：支持多语言（中英文）、三种测试类型并行

## 三种测试类型

| 类型 | 验证什么 | 用什么 |
|------|---------|-------|
| **断言** | 元素是否存在、文本是否正确 | Playwright expect |
| **视觉** | 页面布局有没有变化 | 截图 + pixelmatch 对比 |
| **接口** | UI 显示的数据和 API 是否一致 | GraphQL 查询 |

## 技术亮点（面试官可能会问的）

### 1. Page Object Model

```
页面 = 定位器（元素在哪）+ 操作方法（能做什么）+ 验证方法（结果应该怎样）
```

好处：测试用例只写业务逻辑，不关心具体定位器。

### 2. 多语言支持

- 配置按语言分离：`zh_CN.yaml`、`en_US.yaml`
- 通过 `add_init_script` 提前注入语言设置
- 页面对象从配置读取期望值

### 3. 特殊交互处理（滑块验证码）

Playwright 的 `drag_to` 不够用，模拟完整鼠标事件序列：
```
mousedown → 30步 mousemove → mouseup
```

### 4. Token 管理

GraphQL 客户端实现 Token 缓存，过期自动重新登录，避免每次重复登录。

## 可能遇到的问题

| 问题 | 回答要点 |
|------|---------|
| 为什么选 Playwright？ | 自动等待、内置截图/trace、API 现代 |
| 为什么分三种测试类型？ | 断言验证功能、视觉验证布局、接口验证数据 |
| 配置为什么分离？ | 测试数据可变、代码逻辑不变 |
| 遇到什么难点？ | 滑块验证码、多语言注入、Token 管理 |
| 怎么维护定位器？ | 从 YAML 配置读取，不硬编码在代码里 |
